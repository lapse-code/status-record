# Coding Standards

## 基本原则

- 优先保持简单、可读、可测试。
- 业务规则不要散落在 UI 组件里。
- 所有统计数字都应能追溯到原始记录。
- 标签、状态、不专注原因都应数据驱动，不写死在组件中。内部字段和标签类型仍可沿用 `blocker`，但用户界面应显示“不专注原因”。

## TypeScript

- 开启 `strict`。
- 公共类型放在明确的 domain/types 模块中。
- 避免 `any`；必须使用时说明原因。
- 时间、日期、时长使用专门类型别名或命名清晰的字段。
- 输入数据进入系统边界时做校验。

## 命名

- 文件名使用 kebab-case。
- React 组件使用 PascalCase。
- 函数和变量使用 camelCase。
- 数据库字段使用 snake_case。
- 布尔字段使用 `is_`、`has_`、`can_` 等明确前缀。

## 时间和统计

- 数据库存 UTC timestamp。
- 按天归属的源记录必须同时保存 `local_date` 和 IANA `time_zone`。
- 新记录使用当前设备时区；旧数据或旧备份缺少时区时按 `Asia/Tokyo` 兼容。
- 历史时间线按记录发生时区绘制，不要用当前设备时区重新解释旧记录。
- 时长统一存数字，不存 `"25m"` 这种字符串。
- 统计函数应尽量是纯函数，输入记录数组，输出聚合结果。
- 专注、不专注、拖延统计必须从日点阵 `durationMsByState` 的真实时长汇总得到，不用主色点数量；时间占比只用专注、不专注、拖延做分母，休息不进入分母。
- `updated_at` 是元数据时间，只用于记录更新、导入合并和未来同步；点阵、统计、拖延保护和手动记录重叠判断必须使用业务时间字段，不能用 `updated_at` 推断离岗或专注结束。
- 不要把缓存 rollup 当作唯一真相。

## UI 约定

- 第一屏应是可直接使用的 Today 页面，而不是营销首页。
- 番茄钟主操作必须明显：到岗、开始、暂停、继续、结束复盘。
- 倒计时结束后必须引导复盘，不应静默结束。
- 标签选择支持键盘和鼠标。
- 重要数字要带单位，例如 `25 分钟`、`3 次`。
- 表单保存后给出清楚反馈。
- 全局数据操作入口（示例数据、清除示例、导入、导出）不能只依附于侧边栏；任何会隐藏侧边栏操作区的响应式状态，都必须在主内容区域提供替代入口。

## 数据修改

- 默认使用软删除，避免历史统计断裂。
- 修改标签名不应破坏历史记录。
- 删除标签时应先检查历史引用；已有历史记录的标签只能归档，不能删除。
- 归档标签不应出现在新的复盘选择中，但历史统计必须保留。
- `status-completed` 是专注判定绑定标签，不能归档或删除；不要用标签名称判断专注状态。
- sleep log 同一天重复提交应更新，不新增重复记录。

## 错误处理

- 计时器运行中刷新页面时，应能恢复可解释状态。
- 本地数据库失败时应显示可读错误。
- 导出失败不能损坏本地数据。
- 导入失败不能先清空或损坏本地数据；必须先解析和基础校验，再在事务中写入。
- 统计遇到缺失 review 时应明确排除或显示待复盘，而不是吞掉。

## 文档同步

以下变更必须同步更新文档：

- 数据库 schema 变化。
- 复盘字段变化。
- 休息余额规则变化。
- 默认标签变化。
- 统计口径变化。
