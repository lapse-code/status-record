# Decision Log

## 2026-06-11: 先做本地优先网页应用

决定：第一版实现为网页应用，本地保存数据。

原因：

- 用户希望先用起来验证流程。
- 网页开发速度快，便于调整记录口径。
- 后续可以包装成 Mac app，或评估 iOS/iPad 方案。

影响：

- MVP 不依赖云服务。
- 必须提供数据导出。
- 数据层需要为未来迁移保留边界。

## 2026-06-11: 启动延迟通过到岗时间自动计算

决定：不让用户手填启动延迟。用户到工位后点击到岗，系统记录 `arrived_at`，第一轮番茄钟开始时计算延迟。若用户未提前到岗而直接开始番茄钟，系统会把开始番茄钟的时刻同步记录为到岗时刻，并把启动延迟记为 0。

原因：

- 用户担心手填会美化自己。
- 时间戳比回忆更可靠。

影响：

- 需要 Arrival 模块。
- 如果用户忘记点击到岗但直接开始番茄钟，系统无法恢复此前真实拖延，只能从开始番茄钟这一刻同步到岗并记录 0 分钟启动延迟。

## 2026-06-11: MVP 记录注意力切换次数

决定：暂不记录“深度工作分钟”，改为每轮结束输入注意力切换次数。

原因：

- 深度工作分钟很难自我准确估算。
- 切换次数更容易回忆，也更适合长期观察趋势。

影响：

- 统计中使用注意力切换总次数和每小时切换次数。
- 后续如果接入自动检测，再扩展深度工作分钟。

## 2026-06-11: 状态、产物、阻塞都使用标签系统

决定：五种结束状态、可见产物、主要阻塞都通过 labels 管理。

原因：

- 用户希望数据化，同时保留扩展能力。
- 标签能支持统计图，也能适应后续习惯变化。

影响：

- 不把状态写死成不可变 enum。
- 需要标签管理页面或设置区。

## 2026-06-11: 每 25 分钟学习获得 5 分钟休息余额

决定：完成学习后按 `floor(actual_duration_minutes / 25) * 5` 增加休息余额。

原因：

- 用户不一定每 25 分钟立刻休息。
- 余额机制比强制休息更符合实际使用。

影响：

- 需要 Break Bank 模块。
- 复盘时询问立即使用还是保存。

## 2026-06-11: 提前完成按实际有效专注分钟记录

决定：手动提前点击完成时，`actual_duration_minutes` 按 `completed_at - started_at - paused_total_seconds - active_pause_seconds` 计算，向下取整到分钟，并且不超过计划时长。

原因：

- 避免刚开始就点击完成却记录完整计划时长。
- 休息余额必须跟真实有效学习分钟绑定。

影响：

- 自然倒计时结束仍记录计划时长。
- 暂停中的完成不会把暂停时间计入学习时长。

## 2026-06-11: 睡眠每天只记录一次

决定：睡眠记录只包含睡眠时长和主观精力 1-5，同一天重复输入时修改旧记录。

原因：

- 用户明确不想输入入睡和起床时间。
- 降低每日记录成本。

影响：

- `sleep_logs.local_date` 需要唯一约束。
- 统计以日为粒度。

## 2026-06-11: 初始化 Web MVP 技术栈

决定：使用 React + TypeScript + Vite 实现第一版网页应用，IndexedDB 通过 Dexie 管理，图表使用 Recharts。

原因：

- 当前项目不需要后端。
- 本地优先更适合个人私密记录。
- Vite 开发速度快，React 组件适合快速迭代复盘和统计界面。

影响：

- 项目依赖安装在本项目 `node_modules/`，不需要全局安装 Vite。
- 数据默认保存在浏览器本地，需要通过导出功能备份。
- 后续如加同步，应通过 service/repository 边界替换存储层。

## 2026-06-11: E2E 使用系统 Chrome

决定：Playwright 主流程测试使用本机 Google Chrome channel。

原因：

- 本机已安装 Google Chrome。
- 避免下载 Playwright 自带 Chromium。

影响：

- 本机运行 `npm run e2e` 需要有 Google Chrome。
- CI 环境如果没有系统 Chrome，需要改回 Playwright 托管浏览器并安装浏览器依赖。

## 2026-06-11: 睡眠录入改为 15 分钟步进控件

决定：睡眠录入界面不再使用小数小时，也不再拆成两个数字输入或下拉框；改为 `6:00`、`6:15`、`6:30` 这种 15 分钟粒度的步进输入，默认 `7:00`。上箭头减少 15 分钟，下箭头增加 15 分钟，底层仍保存 `sleep_duration_minutes`。精力评分也使用同样方向的步进输入，范围 1-5。

原因：

- `6.75 小时` 这种显示不符合日常理解。
- 用户更容易用固定步长微调，也可以直接键盘输入。

影响：

- 历史数据无需迁移。
- 历史值如果不是 15 分钟整，界面显示时会对齐到最近的 15 分钟。
- 统计口径不变。

## 2026-06-11: 增加示例数据按钮

决定：在右上角增加“示例数据”按钮，向当前浏览器 IndexedDB 写入 2026-06-01 到 2026-06-10 的 demo 记录。

原因：

- 用户需要不用手动录入就能查看统计页效果。
- 数据保存在浏览器本地，必须由应用自身写入当前浏览器上下文。

影响：

- 重复点击会先清理 `demo-` 前缀记录，再重新写入 demo 数据。
- 不删除用户真实记录。

## 2026-06-11: 记录明细跟随状态、产物、阻塞图筛选

决定：统计页中“状态分布”“产物标签”“阻塞原因”的饼图扇区和右侧标签列表都可以点击，并驱动下方“记录明细”的筛选。

原因：

- 底部区域不应只服务可见产物，也应能回看阻塞原因和状态对应的复盘记录。
- 用户需要从任意分布图里的某个类别直接切到对应记录，而不是手动在长列表里找。

影响：

- 点击状态、产物或阻塞标签后，下方记录只显示该标签关联的复盘记录。
- 切换到其他标签会立即切换筛选。
- 每条记录同时展示状态、产物、阻塞、学习时长、注意力切换次数和可用文本备注。

## 2026-06-11: 增加休息倒计时和休息结束提醒

决定：复盘中的休息选项改为“保留休息”和“使用休息”。选择使用休息后创建 `break_sessions` 记录并启动休息倒计时；休息自然结束或提前结束后，系统重新打开一个 arrival session，用于记录下一轮启动延迟。

原因：

- 用户需要真正的休息倒计时和休息结束提醒，而不只是余额数字。
- 休息时间不应该计入下一轮启动延迟。
- 提前结束休息时，未用完的休息分钟应回到余额，避免因为提前继续学习而损失休息额度。

影响：

- 新增 `break_sessions` 本地表，Dexie schema 升级到 version 2。
- `break_bank_transactions` 仍是休息余额的真相来源。
- 休息结束后，用户可以在番茄钟区选择下一轮固定或自定义时长继续。

## 2026-06-11: 增加日点阵统计图

决定：统计页增加“日点阵”，一天固定 288 个点，每 5 分钟一个点。纵向每列 12 个点代表 1 小时，横向 24 列代表一天，时间顺序为从上到下，再进入下一列。

原因：

- 用户需要用类似方框点阵的方式快速看到一天中延迟、休息、学习和阻塞分布。
- 5 分钟粒度足够细，又不会像分钟级图表一样过密。

影响：

- 红色表示启动延迟。
- 天蓝色表示休息。
- 绿色表示正常学习。
- 黄色表示有阻塞或被打断的学习。
- 周/月视图中，点阵显示当前统计范围内最近有记录的一天；日视图显示当天。

## 2026-06-11: 日点阵支持历史日期导航

决定：日点阵增加前一天、后一天、今天和日期输入控件。默认日期仍按当前统计范围内最近有记录的一天选择；用户手动选择后，可以持续往昨天、前天或更早日期查看。

原因：

- 点阵的价值在于日复盘，必须能连续查看历史日期，而不只是当前统计范围自动选出的某一天。
- 日期输入能快速跳到任意一天，前后按钮适合连续浏览。

影响：

- 点阵日期可以独立于统计页的日/周/月范围。
- 没有记录的日期仍显示完整 288 个空白点。

## 2026-06-11: 增加版本化 JSON 备份导入

决定：导出 JSON 改为版本化备份格式 `status-record.backup` version 1，并增加右上角“导入”按钮。导入支持当前备份格式，也兼容早期直接导出的 `AppSnapshot` 结构。

原因：

- 网页部署后，每个用户的数据仍保存在自己的浏览器 IndexedDB 中，必须有可恢复的备份路径。
- 未来如果增加账号系统，同一份格式可以作为本地数据上传云端的同步包基础。
- 备份结构使用数据库表名，比 UI 内部 camelCase snapshot 更稳定。

影响：

- 导入采用合并写入：同 `id` 更新，不同 `id` 新增；不会先清空本地数据。
- 导入前会检查每张表是数组，并检查主键字段。
- `sleep_logs.local_date` 有唯一约束；如果同一天已有不同 `id` 的睡眠记录，导入时保留本地 id 并写入导入内容。
- Playwright 已覆盖导入备份后统计页出现导入记录。

## 2026-06-11: 使用 GitHub Pages 部署静态网页应用

决定：第一版线上部署使用 GitHub Pages + GitHub Actions。`main` 分支 push 后，workflow 会安装依赖、运行单元测试、lint、构建 Vite `dist`，再通过 Pages artifact 发布。

原因：

- 当前应用是纯前端本地优先应用，不需要后端服务即可给朋友使用。
- 仓库已经在 GitHub，GitHub Pages 能减少额外平台账号和部署链路。
- 构建前跑测试和 lint，可以避免明显坏版本自动上线。

影响：

- Vite 在 `GITHUB_PAGES=true` 时使用 `/status-record/` 作为 `base`，适配仓库 Pages 子路径。
- 线上地址为 `https://lapse-code.github.io/status-record/`。
- 线上用户数据仍保存在各自浏览器 IndexedDB；同步仍依赖导出/导入，尚无账号系统。

## 2026-06-11: 未到岗直接开始番茄钟时自动到岗

决定：`startFocusTimer` 不再允许产生没有到岗记录的专注。用户可以先点“到岗”再开始，也可以直接点番茄钟开始；如果没有打开的 arrival session，系统会同步创建 arrival session，并把本轮 focus session 绑定到它。

原因：

- 用户的工作逻辑是“到岗后才能工作”，不能出现没有到岗上下文的专注记录。
- 直接点击番茄钟开始应被理解为“现在已经到岗并马上开始”，此时启动延迟为 0。

影响：

- 新增 E2E 覆盖“未到岗直接开始番茄钟”的路径。
- 启动延迟统计仍从 arrival 到第一轮 focus start 计算；自动到岗和开始时间相同，因此延迟为 0。

## 2026-06-12: 主界面采用 Focus Studio 布局

决定：主界面从早期默认仪表盘样式改为 Focus Studio 方向：左侧导航和本地数据操作，主工作区优先展示专注倒计时、今日摘要、睡眠、日点阵和最近记录。

原因：

- 用户希望日常使用不累，界面更像安静的个人学习工作台。
- 早期布局存在块不够整齐、空白不自然、点阵轴文字压得太紧等问题。
- Today 页面应服务真实操作，不应像营销页或松散卡片集合。

影响：

- Today 页面上方固定为专注倒计时和今日侧栏，下方用完整宽度展示日点阵和最近记录。
- Today 最近记录默认显示全部内容，不再提供状态、产物、阻塞标签快速筛选；分类查看集中放在统计页的图表和记录明细中。
- 视觉风格保持克制的工作台色彩和 8px 以内圆角。

## 2026-06-12: 日点阵根据容器宽度自适应分列

决定：日点阵的源数据仍保持一天 288 个 5 分钟点，但 UI 根据容器宽度切换展示密度：宽容器每列 30 分钟，窄容器每列 1 小时。

原因：

- 统计页宽屏下使用固定 24 列会留下大量空白。
- 窄容器下压缩 48 列会让小时数字不可读。
- 自适应分列能在 Today 和 Analytics 两个不同容器里保持可读性。

影响：

- 点阵状态颜色和统计口径保持同一套定义：红色为启动延迟，天蓝色为休息，绿色为正常学习，黄色为有阻塞或被打断的学习。
- 日期导航仍支持前一天、后一天、今天和直接输入日期。
- E2E 对点阵说明文案兼容“每列 30 分钟”和“每列 1 小时”两种结果。

## 2026-06-12: 倒计时结束触发浏览器通知和默认提示音

决定：番茄钟自然结束和休息倒计时自然结束时，系统同时触发浏览器通知、内置短提示音和页面内提示。

原因：

- 用户需要在页面不一定处于注意力中心时也能收到提醒。
- 番茄钟结束需要回到复盘，休息结束需要回到下一轮启动延迟和专注选择。
- 不引入自定义音频文件，先使用浏览器通知和 Web Audio 生成的短提示音。

影响：

- `src/reminders.ts` 封装提醒逻辑，主界面只在计时自然结束时调用。
- 用户手动点击“完成”或“提前结束休息”不会触发系统通知，避免误提醒。
- 如果浏览器通知权限不可用或被拒绝，页面内提示和复盘弹窗仍然生效。

## 2026-06-12: 日点阵格子颜色改为多数分钟决定

决定：日点阵仍然显示 5 分钟一个格子，但每个格子内部先按 1 分钟粒度累计状态，再用多数分钟决定最终颜色。只有在分钟数打平时，才使用当前点阵状态优先级决定颜色。

原因：

- 旧逻辑是“只要有重叠就按优先级覆盖”，会让 4 分钟延迟 + 1 分钟学习的格子显示成学习，不符合直觉。
- 多数分钟规则更接近用户对这个 5 分钟区间主要状态的理解。

影响：

- 1 分钟红色延迟 + 4 分钟绿色学习会显示绿色。
- 4 分钟红色延迟 + 1 分钟绿色学习会显示红色。
- 2 分钟绿色学习 + 2 分钟黄色阻塞 + 1 分钟空白会显示黄色。

## 2026-06-12: 休息自然结束后可继续使用剩余余额

决定：休息倒计时自然结束后，如果仍有可用休息余额，不再直接进入下一轮启动延迟，而是弹出“休息结束”选择。用户可以开始学习、继续休息 5 分钟、使用全部剩余休息，或输入自定义休息分钟。

原因：

- 用户可能先使用 5 分钟休息，但手上仍有可用余额。
- 休息结束后是否继续休息应由用户当下判断，而不是系统自动进入学习流程。
- “使用全部”按钮可以快速使用当前最长可用休息时间。

影响：

- `startBreakTimer` 支持复盘之外继续启动休息倒计时。
- 继续休息的最大分钟数是当前休息余额。
- 没有剩余休息余额时，休息结束仍直接进入下一轮启动延迟记录。

## 2026-06-12: 新增周点阵页面

决定：在主导航中新增“周点阵”页面，按周一到周日展示 7 天日点阵，并提供上一周、下一周、本周和日期跳转。

原因：

- 单日点阵适合当天复盘，但用户需要横向比较一周内每天的时间分布。
- 现有 `buildDayTimeline` 已经稳定，周视图可以复用同一套颜色和多数分钟规则。

影响：

- 不新增数据库表，也不影响导入导出格式。
- 周视图只负责展示和日期导航，源数据仍来自现有到岗、专注、复盘和标签记录。

## 2026-06-12: 日点阵加入休息状态并扩大启动延迟覆盖

决定：日点阵新增 `break` 状态，用天蓝色展示 `break_sessions` 记录的休息时间；arrival 区间内未被休息或专注覆盖的等待时间显示为红色启动延迟。

原因：

- 休息和启动延迟都不是学习，但含义完全不同，不能共用红色。
- 用户到岗后或休息结束后还没开始下一轮专注时，应被看作启动延迟。
- 休息中不应被误算成启动延迟，这与 `break_sessions` 的设计一致。

影响：

- 日点阵和周点阵图例新增“休息”计数。
- 多数分钟打平时的颜色优先级更新为 `blocked > focus > startup_delay > break > empty`。
- 不新增数据库表，也不改变导入导出格式；休息状态直接来自现有 `break_sessions`。
