# Decision Log

## 2026-06-11: 先做本地优先网页应用

决定：第一版实现为网页应用，本地保存数据。

原因：

- 用户希望先用起来验证流程。
- 网页开发速度快，便于调整记录口径。
- 后续可以包装成 Mac app，或评估 iOS/iPad 方案。

影响：

- MVP 不依赖云服务。
- 必须提供数据导出。
- 数据层需要为未来迁移保留边界。

## 2026-06-11: 拖延通过到岗时间自动计算

决定：不让用户手填拖延。用户到工位后点击到岗，系统记录 `arrived_at`，第一轮番茄钟开始时计算拖延。若用户未提前到岗而直接开始番茄钟，系统会把开始番茄钟的时刻同步记录为到岗时刻，并把拖延记为 0。内部字段仍沿用 `startup_delay`。

原因：

- 用户担心手填会美化自己。
- 时间戳比回忆更可靠。

影响：

- 需要 Arrival 模块。
- 如果用户忘记点击到岗但直接开始番茄钟，系统无法恢复此前真实拖延，只能从开始番茄钟这一刻同步到岗并记录 0 分钟拖延。

## 2026-06-11: MVP 记录注意力切换次数

决定：暂不记录“深度工作分钟”，改为每轮结束输入注意力切换次数。

原因：

- 深度工作分钟很难自我准确估算。
- 切换次数更容易回忆，也更适合长期观察趋势。

影响：

- 统计中使用注意力切换总次数和每小时切换次数。
- 后续如果接入自动检测，再扩展深度工作分钟。

## 2026-06-11: 状态、产物、不专注原因都使用标签系统

决定：五种结束状态、可见产物、不专注原因都通过 labels 管理。内部标签类型仍沿用早期的 `blocker` 命名。

原因：

- 用户希望数据化，同时保留扩展能力。
- 标签能支持统计图，也能适应后续习惯变化。

影响：

- 不把状态写死成不可变 enum。
- 需要标签管理页面或设置区。

## 2026-06-11: 每 25 分钟专注获得 5 分钟休息余额

状态：已被 2026-06-12 的“休息余额改为每日学习累计账本”替代。此条保留为历史决策记录。

决定：完成专注后按 `floor(actual_duration_minutes / 25) * 5` 增加休息余额。

原因：

- 用户不一定每 25 分钟立刻休息。
- 余额机制比强制休息更符合实际使用。

影响：

- 需要 Break Bank 模块。
- 复盘时询问立即使用还是保存。

## 2026-06-11: 提前完成按实际有效专注分钟记录

决定：手动提前点击完成时，`actual_duration_minutes` 按 `completed_at - started_at - paused_total_seconds - active_pause_seconds` 计算，向下取整到分钟，并且不超过计划时长。

原因：

- 避免刚开始就点击完成却记录完整计划时长。
- 休息余额必须跟真实有效专注分钟绑定。

影响：

- 自然倒计时结束仍记录计划时长。
- 暂停中的完成不会把暂停时间计入专注时长。

## 2026-06-11: 睡眠每天只记录一次

决定：睡眠记录只包含睡眠时长和主观精力 1-5，同一天重复输入时修改旧记录。

原因：

- 用户明确不想输入入睡和起床时间。
- 降低每日记录成本。

影响：

- `sleep_logs.local_date` 需要唯一约束。
- 统计以日为粒度。

## 2026-06-11: 初始化 Web MVP 技术栈

决定：使用 React + TypeScript + Vite 实现第一版网页应用，IndexedDB 通过 Dexie 管理，图表使用 Recharts。

原因：

- 当前项目不需要后端。
- 本地优先更适合个人私密记录。
- Vite 开发速度快，React 组件适合快速迭代复盘和统计界面。

影响：

- 项目依赖安装在本项目 `node_modules/`，不需要全局安装 Vite。
- 数据默认保存在浏览器本地，需要通过导出功能备份。
- 后续如加同步，应通过 service/repository 边界替换存储层。

## 2026-06-11: E2E 使用系统 Chrome

决定：Playwright 主流程测试使用本机 Google Chrome channel。

原因：

- 本机已安装 Google Chrome。
- 避免下载 Playwright 自带 Chromium。

影响：

- 本机运行 `npm run e2e` 需要有 Google Chrome。
- CI 环境如果没有系统 Chrome，需要改回 Playwright 托管浏览器并安装浏览器依赖。

## 2026-06-11: 睡眠录入改为 15 分钟步进控件

决定：睡眠录入界面不再使用小数小时，也不再拆成两个数字输入或下拉框；改为 `6:00`、`6:15`、`6:30` 这种 15 分钟粒度的步进输入，默认 `7:00`。底层仍保存 `sleep_duration_minutes`。精力评分也使用同样的步进输入，范围 1-5。

原因：

- `6.75 小时` 这种显示不符合日常理解。
- 用户更容易用固定步长微调，也可以直接键盘输入。

影响：

- 历史数据无需迁移。
- 历史值如果不是 15 分钟整，界面显示时会对齐到最近的 15 分钟。
- 统计口径不变。

## 2026-06-13: 睡眠和精力步进方向按直觉调整

决定：睡眠时长和主观精力步进控件统一改为上箭头增加、下箭头减少。键盘 `ArrowUp` 和 `ArrowDown` 与按钮方向保持一致。

原因：

- 用户反馈原来“上减、下加”不符合日常数字输入直觉。
- 睡眠时长和精力评分都属于数值控件，应采用同一套增减方向，减少误操作。

影响：

- 睡眠时长上箭头增加 15 分钟，下箭头减少 15 分钟。
- 精力上箭头增加 1 分，下箭头减少 1 分，仍限制在 1-5。
- E2E 中睡眠记录流程继续覆盖 15 分钟步进和精力步进。

## 2026-06-11: 增加示例数据操作

决定：增加“示例数据”操作，向当前浏览器 IndexedDB 写入 2026-06-01 到 2026-06-10 的 demo 记录。入口位置随布局变化，大屏可在侧栏，侧栏隐藏时在主内容区域显示。

原因：

- 用户需要不用手动录入就能查看统计页效果。
- 数据保存在浏览器本地，必须由应用自身写入当前浏览器上下文。

影响：

- 重复点击会先清理 `demo-` 前缀记录，再重新写入 demo 数据。
- 不删除用户真实记录。

## 2026-06-11: 记录明细跟随状态、产物、不专注原因图筛选

决定：统计页中“状态分布”“产物标签”“不专注原因”的饼图扇区和右侧标签列表都可以点击，并驱动下方“记录明细”的筛选。

原因：

- 底部区域不应只服务可见产物，也应能回看不专注原因和状态对应的复盘记录。
- 用户需要从任意分布图里的某个类别直接切到对应记录，而不是手动在长列表里找。

影响：

- 点击状态、产物或不专注原因标签后，下方记录只显示该标签关联的复盘记录。
- 切换到其他标签会立即切换筛选。
- 每条记录同时展示状态、产物、不专注原因、专注时长、注意力切换次数和可用文本备注。

## 2026-06-11: 增加休息倒计时和休息结束提醒

决定：复盘中的休息选项改为“保留休息”和“使用休息”。选择使用休息后创建 `break_sessions` 记录并启动休息倒计时；休息自然结束或提前结束后，系统重新打开一个 arrival session，用于记录下一轮拖延。

原因：

- 用户需要真正的休息倒计时和休息结束提醒，而不只是余额数字。
- 休息时间不应该计入下一轮拖延。
- 提前结束休息时，未用完的休息分钟应回到余额，避免因为提前继续学习而损失休息额度。

影响：

- 新增 `break_sessions` 本地表，Dexie schema 升级到 version 2。
- 此处的 `break_bank_transactions` 真相来源口径已被 2026-06-12 的每日学习累计账本替代；现在新赚取的休息从当天 focus sessions 动态计算，transactions 只记录使用和退回。
- 休息结束后，用户可以在番茄钟区选择下一轮固定或自定义时长继续。

## 2026-06-11: 增加日点阵统计图

决定：统计页增加“日点阵”，一天固定 288 个点，每 5 分钟一个点。初始形态为纵向每列 12 个点代表 1 小时，横向 24 列代表一天，时间顺序为从上到下，再进入下一列；后续已改为根据容器宽度自适应每列 30 分钟或 1 小时。

原因：

- 用户需要用类似方框点阵的方式快速看到一天中拖延、休息、专注和不专注分布。
- 5 分钟粒度足够细，又不会像分钟级图表一样过密。

影响：

- 红色表示拖延。
- 天蓝色表示休息。
- 绿色表示专注。
- 黄色表示不专注。
- 周/月视图中，点阵显示当前统计范围内最近有记录的一天；日视图显示当天。

## 2026-06-11: 日点阵支持历史日期导航

决定：日点阵增加前一天、后一天、今天和日期输入控件。默认日期仍按当前统计范围内最近有记录的一天选择；用户手动选择后，可以持续往昨天、前天或更早日期查看。

原因：

- 点阵的价值在于日复盘，必须能连续查看历史日期，而不只是当前统计范围自动选出的某一天。
- 日期输入能快速跳到任意一天，前后按钮适合连续浏览。

影响：

- 点阵日期可以独立于统计页的日/周/月范围。
- 没有记录的日期仍显示完整 288 个空白点。

## 2026-06-11: 增加版本化 JSON 备份导入

决定：导出 JSON 改为版本化备份格式 `status-record.backup` version 1，并增加“导入”操作。导入支持当前备份格式，也兼容早期直接导出的 `AppSnapshot` 结构。

原因：

- 网页部署后，每个用户的数据仍保存在自己的浏览器 IndexedDB 中，必须有可恢复的备份路径。
- 未来如果增加账号系统，同一份格式可以作为本地数据上传云端的同步包基础。
- 备份结构使用数据库表名，比 UI 内部 camelCase snapshot 更稳定。

影响：

- 导入采用合并写入：同 `id` 更新，不同 `id` 新增；不会先清空本地数据。
- 导入前会检查每张表是数组，并检查主键字段。
- `sleep_logs.local_date` 有唯一约束；如果同一天已有不同 `id` 的睡眠记录，导入时保留本地 id 并写入导入内容。
- Playwright 已覆盖导入备份后统计页出现导入记录。

## 2026-06-11: 使用 GitHub Pages 部署静态网页应用

决定：第一版线上部署使用 GitHub Pages + GitHub Actions。`main` 分支 push 后，workflow 会安装依赖、运行单元测试、lint、构建 Vite `dist`，再通过 Pages artifact 发布。

原因：

- 当前应用是纯前端本地优先应用，不需要后端服务即可给朋友使用。
- 仓库已经在 GitHub，GitHub Pages 能减少额外平台账号和部署链路。
- 构建前跑测试和 lint，可以避免明显坏版本自动上线。

影响：

- Vite 在 `GITHUB_PAGES=true` 时使用 `/status-record/` 作为 `base`，适配仓库 Pages 子路径。
- 线上地址为 `https://lapse-code.github.io/status-record/`。
- 线上用户数据仍保存在各自浏览器 IndexedDB；同步仍依赖导出/导入，尚无账号系统。

## 2026-06-11: 未到岗直接开始番茄钟时自动到岗

决定：`startFocusTimer` 不再允许产生没有到岗记录的专注。用户可以先点“到岗”再开始，也可以直接点番茄钟开始；如果没有打开的 arrival session，系统会同步创建 arrival session，并把本轮 focus session 绑定到它。

原因：

- 用户的工作逻辑是“到岗后才能工作”，不能出现没有到岗上下文的专注记录。
- 直接点击番茄钟开始应被理解为“现在已经到岗并马上开始”，此时拖延为 0。

影响：

- 新增 E2E 覆盖“未到岗直接开始番茄钟”的路径。
- 拖延统计仍从 arrival 到第一轮 focus start 计算；自动到岗和开始时间相同，因此拖延为 0。内部字段仍沿用 `startup_delay`。

## 2026-06-12: 主界面采用 Focus Studio 布局

决定：主界面从早期默认仪表盘样式改为 Focus Studio 方向：左侧导航和本地数据操作，主工作区优先展示专注倒计时、今日摘要、睡眠、日点阵和最近记录。

原因：

- 用户希望日常使用不累，界面更像安静的个人学习工作台。
- 早期布局存在块不够整齐、空白不自然、点阵轴文字压得太紧等问题。
- Today 页面应服务真实操作，不应像营销页或松散卡片集合。

影响：

- Today 页面上方固定为专注倒计时和今日侧栏，下方用完整宽度展示日点阵和最近记录。
- Today 最近记录默认显示全部内容，不再提供状态、产物、不专注原因标签快速筛选；分类查看集中放在统计页的图表和记录明细中。
- 视觉风格保持克制的工作台色彩和 8px 以内圆角。

## 2026-06-12: 日点阵根据容器宽度自适应分列

决定：日点阵的源数据仍保持一天 288 个 5 分钟点，但 UI 根据容器宽度切换展示密度：宽容器每列 30 分钟，窄容器每列 1 小时。

原因：

- 统计页宽屏下使用固定 24 列会留下大量空白。
- 窄容器下压缩 48 列会让小时数字不可读。
- 自适应分列能在 Today 和 Analytics 两个不同容器里保持可读性。

影响：

- 点阵状态颜色和统计口径保持同一套定义：红色为拖延，天蓝色为休息，绿色为专注，黄色为不专注。
- 日期导航仍支持前一天、后一天、今天和直接输入日期。
- E2E 对点阵说明文案兼容“每列 30 分钟”和“每列 1 小时”两种结果。

## 2026-06-12: 倒计时结束触发浏览器通知和默认提示音

决定：番茄钟自然结束和休息倒计时自然结束时，系统同时触发浏览器通知、内置短提示音和页面内提示。

原因：

- 用户需要在页面不一定处于注意力中心时也能收到提醒。
- 番茄钟结束需要回到复盘，休息结束需要回到下一轮拖延和专注选择。
- 不引入自定义音频文件，先使用浏览器通知和 Web Audio 生成的短提示音。

影响：

- `src/reminders.ts` 封装提醒逻辑，主界面只在计时自然结束时调用。
- 用户手动点击“完成”或“提前结束休息”不会触发系统通知，避免误提醒。
- 如果浏览器通知权限不可用或被拒绝，页面内提示和复盘弹窗仍然生效。

## 2026-06-12: 日点阵格子颜色改为多数分钟决定

决定：日点阵仍然显示 5 分钟一个格子，但每个格子内部先按 1 分钟粒度累计状态，再用多数分钟决定最终颜色。只有在分钟数打平时，才使用当前点阵状态优先级决定颜色。

原因：

- 旧逻辑是“只要有重叠就按优先级覆盖”，会让 4 分钟拖延 + 1 分钟专注的格子显示成专注，不符合直觉。
- 多数分钟规则更接近用户对这个 5 分钟区间主要状态的理解。

影响：

- 1 分钟红色拖延 + 4 分钟绿色专注会显示绿色。
- 4 分钟红色拖延 + 1 分钟绿色专注会显示红色。
- 2 分钟绿色专注 + 2 分钟黄色不专注 + 1 分钟空白会显示黄色。

## 2026-06-12: 休息自然结束后可继续使用剩余余额

决定：休息倒计时自然结束后，如果仍有可用休息余额，不再直接进入下一轮拖延，而是弹出“休息结束”选择。用户可以开始专注、继续休息 5 分钟、使用全部剩余休息，或输入自定义休息分钟。

原因：

- 用户可能先使用 5 分钟休息，但手上仍有可用余额。
- 休息结束后是否继续休息应由用户当下判断，而不是系统自动进入学习流程。
- “使用全部”按钮可以快速使用当前最长可用休息时间。

影响：

- `startBreakTimer` 支持复盘之外继续启动休息倒计时。
- 继续休息的最大分钟数是当前休息余额。
- 没有剩余休息余额时，休息结束仍直接进入下一轮拖延记录。

## 2026-06-12: 新增周点阵页面

决定：在主导航中新增“周点阵”页面，按周一到周日展示 7 天日点阵，并提供上一周、下一周、本周和日期跳转。

原因：

- 单日点阵适合当天复盘，但用户需要横向比较一周内每天的时间分布。
- 现有 `buildDayTimeline` 已经稳定，周视图可以复用同一套颜色和多数分钟规则。

影响：

- 不新增数据库表，也不影响导入导出格式。
- 周视图只负责展示和日期导航，源数据仍来自现有到岗、专注、复盘和标签记录。

## 2026-06-12: 日点阵加入休息状态并扩大拖延覆盖

决定：日点阵新增 `break` 状态，用天蓝色展示 `break_sessions` 记录的休息时间；arrival 区间内未被休息或专注覆盖的等待时间显示为红色拖延。内部状态名仍为 `startup_delay`。

原因：

- 休息和拖延都不是专注，但含义完全不同，不能共用红色。
- 用户到岗后或休息结束后还没开始下一轮专注时，应被看作拖延。
- 休息中不应被误算成拖延，这与 `break_sessions` 的设计一致。

影响：

- 日点阵和周点阵图例新增“休息”计数。
- 多数分钟打平时的颜色优先级更新为 `blocked > focus > startup_delay > break > empty`。
- 不新增数据库表，也不改变导入导出格式；休息状态直接来自现有 `break_sessions`。

## 2026-06-12: 用户可见术语改为拖延、专注、不专注

决定：点阵和统计中的用户可见术语改为红色“拖延”、绿色“专注”、黄色“不专注”、天蓝色“休息”。原先的“阻塞原因”在界面中改为“不专注原因”。

原因：

- “阻塞”偏工程术语，普通用户不容易立刻理解。
- “专注/不专注/拖延”更接近用户每天复盘时的自然语言。
- 内部字段名和数据结构保持不变，可以继续兼容导入导出、历史数据和代码逻辑。

影响：

- `startup_delay` 内部字段在 UI 中显示为“拖延”。
- `focus` 点阵状态在 UI 中显示为“专注”。
- `blocked` 点阵状态和 `blocker` 标签类型在 UI 中显示为“不专注”或“不专注原因”。

## 2026-06-12: 数据操作入口不能依赖侧栏可见性

决定：示例数据、导入、导出是全局数据操作。大屏布局中可放在侧栏；当侧栏操作区被响应式布局隐藏时，必须在主内容顶部显示同等入口。

原因：

- 用户指出手机只是例子，任何隐藏侧栏的状态都不能让导入导出入口消失。
- 导入和导出关系到数据备份和恢复，不能依附在可能被隐藏的局部导航区域。

影响：

- `DataActions` 在侧栏和主内容备用区域复用。
- 隐藏的文件导入 input 与可见按钮解耦，避免某个布局下按钮可见但导入不可用。
- E2E 新增窄屏场景，验证侧栏入口隐藏时主内容入口仍可见，并且导入功能可用。

## 2026-06-12: 暂停进入拖延并记录专注片段

决定：暂停不是简单冻结倒计时。点击暂停时，系统关闭当前真实专注片段，并让后续时间继续落在打开的到岗记录中，因此点阵显示为拖延；点击继续时，在同一轮番茄钟下开启新的专注片段；点击完成时，按当前有效专注分钟结束本轮并进入复盘；点击取消时，本轮不进入专注统计。

原因：

- 用户期望暂停和完成都会先记录已经学过的时长。
- 暂停和完成的区别是：暂停之后仍可继续同一轮，完成会中断本轮。
- 旧模型只用 `started_at + actual_duration_minutes` 无法准确表示暂停前后多个真实专注区间，会把暂停时间误画成专注。

影响：

- 新增 `focus_segments` 表，当时 Dexie schema 升级到 version 3；后续时区迁移已升级到 version 4。
- 开始番茄钟时创建 running segment；暂停关闭 segment；继续创建新 segment；完成关闭当前 segment。
- 日点阵和周点阵优先使用 completed focus segments；旧数据没有 segments 时回退到旧连续区间。
- JSON 导出包含 `focus_segments`；导入旧备份缺少该表时按空数组兼容。

## 2026-06-12: 不专注原因图表排除“无”并优化窄屏布局

决定：“不专注原因”图表不再把默认“无”显示为一个原因。复盘记录和导入导出仍保留“无”，但图表展示层排除它；如果排除后没有任何原因，显示“暂无不专注原因记录”。统计页图表从固定三列改为自适应列宽，分布图卡片过窄时饼图和标签列表上下排列。

原因：

- “无”表示本轮没有不专注原因，不是一个需要分析的原因类别。
- 半屏或较窄窗口下，固定三列会让饼图和标签互相挤压，影响可读性。

影响：

- `blockerLabelCounts` 底层聚合仍保留“无”，避免破坏历史数据和备份格式。
- 只有 UI 中的“不专注原因”分布图排除“无”。
- 统计页分布图使用自适应布局，标签允许换行，饼图半径随容器缩放。

## 2026-06-12: 开放到岗记录必须保持单例

决定：到岗是一个开放状态。已有未关闭到岗记录时，重复点击到岗或开始番茄钟必须复用已有记录，不得新增 arrival session，也不得重置 `arrived_at`。如果历史数据里已经存在多条未关闭到岗记录，系统以最早打开的那条作为当前有效到岗；点击离开或进入休息时，会把重复开放记录一并关闭。

原因：

- 用户认为到岗时间只能开启一次，关闭前重复开启应无效。
- 如果重复开放记录按最新一条显示，会让 UI 看起来像开始番茄钟重置了到岗时间，拖延统计也会被低估。
- 旧数据或旧版本可能已经留下重复开放记录，需要在读取和关闭时兜底。

影响：

- `startFocusTimer` 绑定最早未关闭 arrival session。
- Today 页面展示最早未关闭 arrival session。
- `checkOutArrival` 和休息切换会关闭全部未关闭 arrival session，恢复“最多一个开放到岗”的状态。
- E2E 覆盖已有到岗开始番茄钟不重置，以及重复开放到岗时绑定最早记录。

## 2026-06-12: 休息余额改为每日学习累计账本

决定：休息奖励不再按单轮 `floor(actual_duration_minutes / 25) * 5` 独立计算，而是按“每日学习累计账本”计算。当天所有已完成番茄钟的真实有效分钟累计到同一个账本，每满 25 分钟获得 5 分钟今日休息余额；未满 25 分钟的进度和未使用休息余额都不跨天保留。`break_bank_transactions` 不再写入新的 `earned` 交易，只记录使用休息和提前结束退回；已赚休息从当天 `focus_sessions.actual_duration_minutes` 动态计算。

原因：

- 用户希望 10 分钟 + 15 分钟也能跨轮累计成 25 分钟并获得 5 分钟休息。
- 用户希望取消当前轮时，本轮不会进入学习累计，也不会产生休息余额。
- 休息余额不应该变成长期账户，第二天应重新开始。

影响：

- `completeFocusTimer` 的 `earned_break_minutes` 表示本轮让今日累计新跨过几个 25 分钟门槛。
- Today 页面显示今日休息余额和 `x / 25` 的今日进度。
- 旧备份中的 `earned` 交易仍保留导入导出；如果没有 focus sessions 可动态计算，会按旧 `earned` 交易兼容。
- 单元测试新增每日账本、跨轮累计和跨天清零覆盖；E2E 新增 15 + 15 分钟跨轮获得休息覆盖。

## 2026-06-12: 点阵实时显示当前进行中的状态

决定：日点阵和周点阵不只读取已复盘的历史结果，也会把当前正在发生的状态纳入计算。打开的到岗记录会延伸到当前 `now` 并显示为拖延；运行中的专注片段会延伸到当前 `now` 和本轮计划专注时长预算两者中更早的时间，并显示为专注；运行中的休息倒计时显示为休息；暂停后只保留暂停前已关闭的专注片段，暂停区间回到拖延；完成但尚未复盘的专注先显示为专注，保存复盘后再按最终标签改为专注或不专注；取消的专注片段不进入点阵。

原因：

- 用户需要点阵反映“此刻系统认为正在发生什么”，而不是等整轮结束和复盘保存后才变化。
- 当前 5 分钟格子的颜色应继续遵守多数分钟规则，例如前 2 分钟拖延、后 3 分钟专注时显示专注。
- 不专注只能在复盘后判断，不能在专注过程中提前预测。
- 取消本轮意味着这段临时专注颜色必须撤销。

影响：

- `buildDayTimeline` 接受 `now`，并纳入 running focus segment、running break session 和开放 arrival session。
- Today 页、统计页日点阵和周点阵都会传入 App 的每秒 `now` 状态。
- 单元测试新增实时拖延、实时专注、暂停拖延、取消撤销、完成未复盘和实时休息覆盖。

## 2026-06-12: 时间线按记录发生时区绘制

决定：按天归属的源记录保存 UTC 时间戳、`local_date` 和 IANA `time_zone`。新记录使用当前设备时区；旧数据或旧备份缺少 `time_zone` 时按 `Asia/Tokyo` 回填。日点阵和周点阵按每条记录自己的 `time_zone` 把 UTC 区间映射到目标日期的本地分钟，不用用户当前设备时区重新解释历史记录。

原因：

- 用户可能旅行或切换设备时区，如果历史 UTC 时间直接按当前时区展示，会把原本正常作息误显示成熬夜。
- `local_date` 只能解决日期归属，不能解决一天内部 00:00-23:59 的分钟位置；点阵需要知道记录发生时区。
- 记录发生时区是最稳定的默认口径，后续仍可增加“按当前位置时区查看”的显示模式。

影响：

- Dexie schema 升级到 version 4，为到岗、专注、分段、休息交易、休息倒计时和睡眠记录增加 `time_zone` 索引并迁移旧数据。
- 导出备份包含 `time_zone`；导入旧备份时自动回填 `Asia/Tokyo`。
- `buildDayTimeline` 按记录时区映射分钟；Today、统计页日点阵和周点阵标题会显示单一时区或“多时区”提示。
- 新增时间函数和单元测试，覆盖 UTC 时间在东京/洛杉矶的日期差异，以及按记录时区放置点阵格子。

## 2026-06-13: 标签页升级为设置页

决定：侧边栏入口从“标签”改为“设置”。设置页顶部管理点阵状态颜色；下方按状态、产物、不专注原因分组管理标签。每条标签默认只显示颜色、名称、正常/已归档状态、历史记录数量和齿轮设置按钮。齿轮弹窗负责改名、改颜色、保存、归档/解除归档和删除。

原因：

- 用户认为原来的“保存/隐藏”行内操作难以理解，且“隐藏”不能准确表达历史保留但新记录不可选的语义。
- 点阵图状态颜色属于系统显示设置，不应该混在新增标签表单里。
- 有历史数据但长期不用的标签需要归档；没有历史价值的标签才适合删除。

影响：

- `labels.is_active = false` 明确表示归档；归档标签不会出现在复盘弹窗中，但历史统计仍然保留。
- `deleteLabel` 会检查 `session_reviews.status_label_id` 和 `session_review_labels.label_id` 引用；已有历史记录的标签不能删除，只能归档。
- 点阵颜色保存到 `app_settings.timelineColors`，并随 JSON 导入导出一起备份。
- E2E 更新为覆盖新增并归档自定义标签。
